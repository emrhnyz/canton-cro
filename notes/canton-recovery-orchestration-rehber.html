Recovery Orchestration

Party / participant taşıma ve felaket kurtarmasını elle 13 adımdan (offline replication baseline), tek komutluk güvenli bir orchestration aracına indiren açık kaynak proje. Bu sayfa vibe-coding ile parçalı ilerlemek için yazıldı - tek prompt’ta her şeyi yaptırma. Yapıma başlamadan önce: DA roadmap sorusu + grant metninde AI-slop bot kuralları rehbere eklendi.

1. Fikri insan gibi anlatan metin

Canton’da her kurum kendi participant’ını (node) çalıştırır; varlıklar ve sözleşmeler party kimliğine bağlıdır. Bir gün node çöker, validator değişir veya ortam taşınırsa operatörün yapması gereken şey belli: güvenli yedekten doğru şekilde ayağa kaldırmak.

Bugün bu süreç çoğu yerde manuel: onlarca doküman satırı, konsol komutu, sırayı kaçırırsan bozulma riski. Özellikle yarım kalan ACS import gibi hatalarda sistem “sessizce yanlış” kalabilir; sonra ACS_COMMITMENT_MISMATCH gibi ağrılı semptomlar çıkar.

Canton Recovery Orchestration bunun için ortak, açık kaynak bir araç olacak: planla → doğrula → çalıştır → hata olursa dur/geri al → tatbikatı CI’da tekrarla. Cüzdan değil, token değil, site değil — validator operatörlerinin kurtarma kumanda çubuğu.

Grant cümlesi (ezberle) “Canton’da party/participant recovery hâlâ manuel; biz key’e dokunmadan dry-run migration, backup doğrulama ve restore drill sunan open-source orchestration CLI getiriyoruz.”
Neden bu?
  • Fon “resilience / operasyonel basitlik / common good” istiyor
  • CWIF gibi key export tuzağı yok
  • DEX/cüzdan/SDK rafı kadar dolu değil
  • Demo net: taşı → bilerek boz → yakala → düzelt
Başarı tanımı
  • LocalNet’te uçtan uca recovery senaryosu tek komutla
  • Dry-run (dokunmadan plan)
  • Idempotent + kaldığı yerden devam (resume)
  • Kasten bozuk restore’u otomatik yakalama
  • CI’da koşan drill

AŞAMA 1 · Hafta 1

Manuel baseline (en kritik aşama)

amaç: acıyı ölç çıktı: 13-adım defteri

Henüz tool yazma. Offline party replication / backup-restore senaryosunu elle bir kez baştan sona yap. Her adım için yaz:

  • Komut / UI / doküman adımı
  • Ön koşul (pre-condition)
  • Son koşul (post-condition) — nasıl “başarılı”yı ölçtün?
  • Yanlış yapılırsa ne olur?
Fail koşulu (fikri öldüren test) Eğer tüm akış mevcut bir bootstrap script’iyle ~20 satırda zaten kolayca çözülüyorsa, otomasyon değeri yok demektir — scope’u yeniden düşün veya bırak.
Bitti kriteri docs/manual-baseline.md dosyası dolu; her adımın pre/post condition’ı makine-okunur (sonraki kod bunları assert edecek).
AŞAMA 2 · Hafta 2

Tek happy-path komutu (iske + resume)

amaç: wrapper değil, orchestration çıktı: cro apply

Ne yazılacak?

  • Adım makinesi (state machine): step-1 … step-N
  • Her adım sonrası state kaydı (hangi adım tamam?)
  • Ortalarda kırılınca resume ile kaldığı yerden devam
  • Aynı komutu ikinci kez çalıştırınca güvenli no-op (idempotent)
Bitti kriteri Happy path LocalNet’te uçtan uca yeşil. Ortada process kill edip resume ile bitirebiliyorsun. README’de 3 dakikalık “manuel vs otomatik” anlatımı var.
AŞAMA 3 · Hafta 3

Dry-run + preflight

amaç: dokunmadan plan çıktı: cro plan / preflight

Preflight örnekleri (dokümana göre uyarlanır):

  • Backup yaşı / tazelik penceresi
  • Onboarding / migration flag’leri
  • Pruning penceresi uygun mu?
  • Package vetting durumu
  • Hedef participant sağlığı

plan komutu hiçbir mutate etmeden “şunu yapacağım” listesini basmalı.

Bitti kriteri Kasıtlı kötü preflight’ta (eski backup vs.) komut net hata veriyor; iyi durumda plan çıktısı okunaklı.
AŞAMA 4 · Hafta 4

Fault drill + CI

amaç: değeri kanıtla çıktı: cro drill + GitHub Action

Bilerek yarım / bozuk ACS import üret. Araç mismatch’i (veya dokümandaki eşdeğer hatayı) yakalasın, net teşhis + güvenli durma / rollback yolu göstersin.

Fail koşulu Mismatch’i deterministik üretemiyor veya yakalayamıyorsan — “console script wrapper” itirazı haklı çıkar. Grant öncesi bunu çöz veya beyanı küçült.
Bitti kriteri CI’da yeşil drill; kısa demo video (2–4 dk); threat-model notu (“key’e dokunmuyoruz”).
